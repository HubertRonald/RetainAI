# Explainability Architecture

**Version:** v0.1  
**Status:** Draft  
**Sprint:** Sprint 5 — Explainability (SHAP)  
**Repository:** RetainAI

---

# 1. Overview

The Explainability layer provides transparent and reproducible interpretation of machine learning predictions generated by RetainAI.

Its primary objective is to transform predictive models into interpretable analytical components suitable for HR decision support.

Rather than improving predictive performance, this layer explains **why** a prediction was produced and identifies the variables that most strongly influence employee attrition risk.

The architecture is intentionally modular so that alternative explainability techniques may be incorporated in future releases without modifying downstream consumers such as dashboards or AI services.

---

# 2. Objectives

The Explainability layer is designed to provide:

- global model interpretation;
- local prediction explanations;
- feature contribution analysis;
- comparison across predictive models;
- reusable explainability artifacts;
- dashboard-ready visualizations;
- future compatibility with AI-assisted recommendation systems.

---

# 3. Scope

RetainAI v0.2 introduces SHAP as the initial explainability framework.

Supported predictive models include:

- Logistic Regression
- Random Forest
- XGBoost

Each model uses the most appropriate SHAP explainer implementation while exposing a common interface to the rest of the system.

---

# 4. Explainability Strategy

Explainability is divided into two complementary perspectives.

## Global Explainability

Global explanations describe overall model behavior across the validation dataset.

Typical outputs include:

- Mean Absolute SHAP Values
- Feature Importance Ranking
- Beeswarm Plot
- Summary Plot

Questions addressed include:

- Which variables consistently influence attrition?
- Which predictors contribute most to model decisions?
- How stable are explanations across different algorithms?

---

## Local Explainability

Local explanations describe individual employee predictions.

Typical outputs include:

- Waterfall Plot
- Force Plot (future)
- Individual Feature Contributions

Questions addressed include:

- Why was this employee classified as high attrition risk?
- Which variables increased or reduced the predicted probability?
- What is the relative importance of each predictor for this specific case?

---

# 5. Model Abstraction

The Explainability layer abstracts the SHAP implementation according to model type.

```text
Logistic Regression
        │
        ▼
LinearExplainer

Random Forest
        │
        ▼
TreeExplainer

XGBoost
        │
        ▼
TreeExplainer
````

Notebook code never instantiates SHAP directly.

Explainer selection is centralized inside the Explainability module.

# 6. Module Architecture

```text
modules/
└── explainability/
    ├── preprocessing.py
    ├── explainers.py
    ├── shap_values.py
    ├── feature_importance.py
    ├── visualization.py
    └── report.py
```

---

## preprocessing.py

Responsible for:

* preparing transformed feature matrices;
* preserving feature names;
* validating feature consistency;
* creating background samples for SHAP.

---

## explainers.py

Factory responsible for selecting the appropriate SHAP explainer implementation according to the trained model.

The remainder of the pipeline remains independent of SHAP internals.

---

## shap_values.py

Responsible for:

* computing SHAP values;
* extracting expected values;
* normalizing outputs across different explainers.

---

## feature_importance.py

Responsible for:

* computing Mean Absolute SHAP values;
* ranking predictors;
* exporting reusable importance tables.

---

## visualization.py

Responsible for generating:

* Feature Importance Plot
* Beeswarm Plot
* Summary Plot
* Waterfall Plot
* Dependence Plot (future)

---

## report.py

Responsible for generating reproducible Markdown reports summarizing explainability results.

# 7. Source Package

```text
src/
└── retainai/
    └── explainability/
        └── run_explainability.py
```

The executable pipeline is responsible for:

* loading trained models;
* loading validation datasets;
* computing SHAP explanations;
* generating explainability artifacts;
* exporting reports and figures.

---

# 8. Generated Artifacts

The Explainability pipeline generates reproducible artifacts that can be consumed by subsequent project layers.

```text
artifacts/
│
├── reports/
│   └── explainability_report_<model>.md
│
├── figures/
│   └── explainability/
│       └── <model>/
│           ├── feature_importance.png
│           ├── beeswarm.png
│           └── waterfall_example.png
│
└── explanations/
    └── <model>/
        ├── shap_values.joblib
        ├── expected_value.joblib
        ├── feature_importance.parquet
        └── metadata.json
```

The serialized explanation artifacts enable downstream reuse without recalculating SHAP values.

---

# 9. Notebook

```text
notebooks/
└── 06_explainability.ipynb
```

The notebook is intentionally declarative.

Its responsibilities are limited to:

* loading reusable modules;
* executing the explainability pipeline;
* validating generated outputs;
* displaying visualizations.

All business logic remains encapsulated inside `modules/explainability`.

---

# 10. Dashboard Integration

The Streamlit dashboard consumes the generated explainability artifacts directly.

This design avoids recalculating SHAP values during interactive sessions and significantly reduces dashboard latency.

---

# 11. Future Evolution

The Explainability architecture serves as the foundation for future AI-assisted recommendation capabilities.

Planned evolution:

```text
SHAP Values
        │
        ▼
Structured Explainability
        │
        ▼
LLM-ready JSON
        │
        ▼
Amazon Bedrock
        │
        ▼
```

Future versions will transform structured SHAP outputs into natural-language recommendations while preserving explainability and auditability.
